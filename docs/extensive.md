# 多聴シェル（Extensive）仕様

アーキテクチャ全体は [architecture.md](./architecture.md)。精聴との思想の対比は [background.md](./background.md) §2 と本書 §1 を参照。

## 1. 設計原則

精聴と**意図的に思想を逆転**する。

| 精聴 | 多聴 |
|---|---|
| 診断精度 > 完遂量 | 接触量 > 診断 |
| Cloze で弱点可視化 | 正答率を測らない |
| リプレイ制限 | リプレイ自由・倍速・連続再生 |
| 摩擦 UI | スワイプ・低摩擦 |

効果は **遭遇量**（累計再生時間・構造の文中出現回数・チャンク遭遇）で測る。

## 2. 機能

### 2.1 セットアップ

| 項目 | 選択肢 |
|---|---|
| CEFR | A1+A2 / B1 / B2 |
| シーン | 電話 / 店舗 / 職場 等 |
| 音韻Lv | 1〜4（多聴 UI） |
| コンテンツ長 | 短いパッセージ（3〜6文）/ 長い（5〜8文）/ 対話（4〜8ターン） |
| 構造フラグ | 関係詞節 / 分詞構文 / 仮定法 / 倒置（任意・複数可） |
| 表示モード | 読みながら聞く / 聞くだけ |

### 2.2 input flooding（核心機能）

構造フラグ ON 時:

1. プロンプトに「**各パッセージの 80% 以上の文**に該当構造を含める」旨と few-shot 例を注入
2. 生成後、パターン検証（`structureValidation.js`）で合格率を判定
3. 不合格なら最大 5 回リトライ。それでも不合格の場合は配信するが、**合格扱いにしない**

開発確認: `?debug=1` でコンソールに検証結果。`node scripts/validate-structure-flooding.mjs` で few-shot 例の検証。

### 2.3 聴取セッション

| 操作 | 動作 |
|---|---|
| 自動 ON/OFF | パッセージ終了後の連続生成・再生 |
| 速度 | 1.0x / 1.25x / 0.85x |
| スワイプ | 前後のパッセージへ |
| シャドーに追加 | 現在パッセージをシャドーキューへ（追加済み表示・満杯時通知） |
| セットアップ | 設定画面に戻る |

**聴取中は Stats を表示しない**（無摩擦原則）。

### 2.4 統計（設定画面のみ）

| 指標 | 意味 |
|---|---|
| 合計分 / パッセージ数 | 再生完了ベースの累計 |
| 構造「接触 N 回」 | 文中の該当構造の**出現回数**の累計（フラグ ON/OFF 問わず自然出現も含む） |
| チャンク遭遇 | `cefr_metadata.used_chunks` の遭遇回数・異なる文脈数（CEFR 切替でフィルタ） |

構造検証合格率はユーザー UI 非表示（`?debug=1` のみ）。

データ: `localStorage`（`elt_extensive_stats`）+ Drive 同期。スキーマ v2。

### 2.5 Past items

聴いたパッセージの履歴。音声はブラウザ + Drive にキャッシュ。

## 3. 意図的にやらないこと

- Cloze / ディクテーション / 正答率
- 聴取中の成績表示
- 層3 の個別診断（精聴の役割）
