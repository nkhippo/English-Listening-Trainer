# English Listening Trainer — 設計思想と問題設計

このドキュメントは、本アプリで解こうとしている課題、それに対する方針、そして問題設計（思想と具体）を一箇所にまとめた**上流設計書**である。コード（実装）と仕様書（`docs/specification.md`）の間に位置し、「なぜそうなっているか」を説明する。

仕様書が **what** を扱うのに対し、本書は **why** を扱う。改修判断に迷ったとき、本書に立ち返ること。

## 1. 解こうとしている課題

### 1.1 前提となる構造理解

リスニングの構造を `english_speech_structure.md`（添付資料）で以下のように整理している：

- **機能軸（5層）**: 語強勢 / 文強勢・リズム / 連結部 / 分節音 / イントネーション
- **物理軸（4パラメータ）**: 調音 A / ピッチ B / 長さ C / 強さ D

このうち、**日本人学習者の最大のボトルネックは層3：連結部（リンキング・弱形・脱落・縮約）**である。理由は3つ：

1. 日本語にはこの現象が構造的に存在しない（モーラ等時リズム＋音節境界が明瞭）ため、入力として知覚するための回路が成人後に獲得困難
2. 「話せないものは聞き取れない」が最も強く当てはまる層であり、産出経験のなさが受容能力を直接制約する
3. 学校英語が層3を体系的に扱わないため、英語学習歴が長くてもこの層だけが置き去りになる

### 1.2 既存アプローチの限界

市販のリスニングアプリ・教材の典型的な構造とその限界：

| アプローチ | 構造 | 限界 |
|---|---|---|
| シャドーイング | 音声を追って復唱 | 産出側の練習。聞き取れているかの診断が困難 |
| 多読・多聴 | 大量の英語音声に触れる | フィードバックループが弱く、どこで聞き落としたか不明 |
| ニュース教材 | 整った発話で語彙拡充 | 層3の出現密度が低く、連結部の訓練価値が小さい |
| 全文ディクテーション | 聞いた英語を全部書き起こす | 綴りミス・既知の内容語のノイズが大きく、層3の診断精度が下がる |
| 4択リスニング問題 | 内容理解の確認 | 推測で正答可能。音韻レベルの聞き取り能力を測れない |

これらに共通する欠点は、**「どの層で・何を聞き落としたか」を構造的に可視化する仕組みがない**こと。学習者は「なんとなく聞き取れない」という曖昧な手応えのまま反復し続けることになる。

### 1.3 本アプリが解こうとする具体的課題

> **層3を主戦場として、聞き落としを構造化されたフィードバックとして返すリスニング訓練の場を提供する。**

副次的に層4（分節音、特にミニマルペア）と層2（文強勢・リズム、弱形復元を通じて間接的に）にも効く設計とする。層1・層5・産出側は本アプリのスコープ外とし、明確に切り捨てる。

## 2. 対応方針

### 2.1 なぜディクテーション（書き起こし）か

ディクテーションは産出を伴わない数少ない訓練法でありながら、**「聞き取った内容を言語化する」という強制機能**を持つ。曖昧な手応えで終わらせず、聞き取りを離散化して判定可能にする。

選択肢式（4択など）は推測で正答できるため診断精度が低い。シャドーイングは産出側に寄りすぎる。ディクテーションは「入力専用かつ判定可能」というスイートスポットにある。

### 2.2 なぜ Cloze（空欄補充）をデフォルトにするか

ディクテーションには2つの形態がある：

- **Full Dictation**: 全文書き起こし
- **Cloze**: 機能語・連結箇所だけを空欄化

Full Dictation は包括的だが、以下の理由で**日常練習には不向き**：

1. 綴りミス・タイポが正答率を押し下げ、層3の診断精度を曇らせる
2. 既知の内容語の入力にかかる認知資源が、層3の音韻判定から注意を奪う
3. 1問あたりの所要時間が長く、反復回数が稼げない

Cloze は **「診断価値の高い場所だけに認知資源を集中させる」** 設計。空欄を `weak_form` `linking` `reduction` `elision` の発生箇所に絞り、内容語は表示する。これにより：

- 1問あたりの所要時間が短縮 → 反復が稼げる
- 失敗箇所が「どの層3現象か」と直結 → 構造化フィードバックが可能
- 内容理解ではなく純粋に音韻に集中できる

Full Dictation はゼロにせず「週次の総合チェック」として保持する。Minimal Pair は層4のピンポイント矯正として用意する。**3モードは目的が異なる併存関係**である。

### 2.3 なぜ3軸設計（モード × シーン × レベル）か

学習進捗を1次元の難易度ですべて表現すると、進捗管理が壊れる。本アプリでは難易度を以下の2軸に分解する：

- **シーン**: 何を話すか（語彙・レジスター・定型句の傾向）
- **レベル**: どう話すか（速度・縮約密度・対話の有無）

同じ「電話口」でも、Lv1 の `Could you hold for a moment, please?`（明瞭・低速・無縮約）と、Lv4 の `Lemme put ya on hold real quick`（自然速・縮約あり）は訓練価値が全く異なる。これを「電話＝中級」「友達会話＝上級」と畳み込むと、

- 学習者は「電話の中級」と「電話の上級」の差を体験できない
- 進捗トラッキングが「電話シーン何問正解」になり、層別の弱点が見えない

モードはさらに直交する第3軸であり、目的が違うため難易度軸とは別管理。

### 2.4 スコープの明確化

| 層 | スコープ | 理由 |
|---|---|---|
| 層1：語強勢 | △ 部分対応 | Cloze 空欄が強勢音節を含む場合は副次的に訓練される。明示的訓練はしない |
| 層2：文強勢・リズム | ◯ 副次的に対応 | 機能語の弱形復元を通じて間接的に訓練 |
| **層3：連結部** | **◎ 主戦場** | 本アプリの存在意義 |
| 層4：分節音 | ◯ Minimal Pair で対応 | 意味を変えるミニマルペアのみ訓練 |
| 層5：イントネーション | × スコープ外 | テキスト書き起こしではピッチ情報が落ち、判定不能 |
| 産出側 | × スコープ外 | 本アプリは入力専用 |

層5と産出側を切り捨てたのは「諦め」ではなく**「他のチャネルで補完すべき」**という判断。具体的には：

- 層5：イントネーション → Pines Academy の 1-on-1、添付の Speaking Training 教材 Part1・2
- 産出側 → Pines Academy の 1-on-1、シャドーイング教材

スコープを絞ることで、本アプリの中で扱う範囲については**深く・構造的に**訓練できる。

## 3. 問題設計の思想

### 3.1 診断精度 > 完遂量

学習アプリの典型的な罠は「問題数を稼ぐこと」を価値だと錯覚すること。本アプリは反対の立場を取る：

> **1問の中で「どこを・どう聞き落としたか」を構造化して返せることが、100問解いて曖昧な手応えを得ることより価値が高い。**

このため、Claude API が文を生成するときに `target_features` を必ず返させ、Review 画面で「弱形を取れたか / 連結を取れたか / 縮約を取れたか」を1問ごとに可視化する。

### 3.2 語彙ではなく音韻で落とす

リスニング失敗の原因は大別して2つ：

1. **語彙不明**: そもそも知らない単語が出てきた
2. **音韻判定不能**: 知っている単語だが音として認識できなかった

本アプリは(2)を訓練する場である。(1)で失敗すると診断ノイズが大きくなるため、**語彙レベルを CEFR A2–B1 に統制する**。Claude への生成プロンプトに語彙制約を強く与えている。

これは「学習者を甘やかしている」のではなく、「層3の訓練に集中するための環境整備」である。語彙拡充は別チャネル（English-Phrase アプリ等）で行う。

### 3.3 フィードバックを構造化する

Naoya 自身の X/Y/Z 文法フレームワークと同じ思想：**「構造として説明できないものは習得できない」**。

リスニング側でも同じ：「なんとなく聞き取れなかった」を「弱形 `was` を取れなかった」「連結 `pick it up` を `picked up` と聞いてしまった」に解像する。これにより：

- 学習者は自分の弱点パターンを認識できる
- 反復の方向性が明確になる（同じパターンを集中的に練習）
- 改善が定量化される（「弱形捕捉率 60% → 80%」のようなトラッキングが可能になる、将来）

### 3.4 制約が学習を作る

無限リプレイ・無限ヒントは学習を阻害する。実力以上のスコアが出てしまい、本人にも「どこまで聞き取れているか」が分からなくなる。本アプリの制約：

- 自然速度の再生のみがデフォルト
- 2回再生失敗で初めて 0.75x スロー再生が解放
- 答えは Check ボタンで一度判定されたら確定（やり直し不可）

これは厳しさのための厳しさではなく、**「自然速度で聞き取れる範囲」を学習者自身に正確に把握させるための診断装置**である。

### 3.5 UI は思想の身体化

本アプリの視覚デザインで唯一の彩度を持つ要素は、**音声再生中だけ青く脈打つ波形ライン**である。それ以外は全て紙のような無彩色で構成される。

これは「聞いている瞬間だけ世界が色づく」という思想の直接的な表現。学習者の注意を「音そのもの」に向けるための設計であり、装飾ではない。

## 4. 問題設計の具体

### 4.1 target_features 分類体系

Claude API が生成する `target_features` は以下の語彙で記述される：

| トークン形式 | 意味 | 例 |
|---|---|---|
| `weak_form:WORD` | 機能語の弱形化 | `weak_form:to` `weak_form:and` `weak_form:was` |
| `linking:WORD1_WORD2` | 子音＋母音のリンキング | `linking:pick_it` `linking:an_apple` |
| `reduction:FORM` | 口語的縮約 | `reduction:gonna` `reduction:wanna` `reduction:didja` |
| `elision:WORD` | 音の脱落 | `elision:next_day`（/t/ が落ちる） |
| `minimal_pair:A_vs_B` | 紛らわしい音 | `minimal_pair:right_vs_light` |

この分類は MECE を厳密には満たさない（`reduction:gonna` は `weak_form:to` の極端形でもある）。しかし**学習者にとって構造化されていれば実用上十分**という立場を取る。完璧な分類体系より、診断が走ることを優先する。

### 4.2 文設計ルール（レベル別）

| Lv | 文長 | 速度 | 縮約 | 機能語密度 | リンキング箇所 | 形態 |
|---|---|---|---|---|---|---|
| 1 | 5–8語 | 0.85x | なし | 低 | 0 | 単文 |
| 2 | 8–10語 | 0.9x | 標準縮約のみ（I'll, don't） | 中 | 自然発生のみ | 単文 |
| 3 | 10–12語 | 1.0x | 標準縮約＋連結 | 中 | 1箇所以上必須 | 単文〜短ターン |
| 4 | 12–16語 | 1.05x | gonna/wanna/lemme/didja | 高 | 2箇所以上必須 | 単文 |
| 5 | 6–14語 × 2–4ターン | 1.05x | フル自然 | 高 | 各ターン必須 | 対話 |

この制約は Claude への生成プロンプト（`src/lib/prompts.js`）に明示的に書き込まれている。改修時はプロンプトを直接編集する。

### 4.3 Cloze 空欄選定の基準

空欄は以下の優先順位で選定する（Claude が自動判定）：

1. **第一優先**: その文に含まれる `target_features` に対応する語・句
   - `weak_form:to` があれば `to` を空欄に
   - `linking:pick_it` があれば `pick it`（あるいは `pick it up`）を空欄に
2. **第二優先**: 機能語（to / of / and / the / for / a / an / can / have / was / will）
3. **回避**: 内容語のうち、CEFR A2 で初出となる語（語彙不明で落とすことを避ける）

空欄数は 2–4 個に制限。多すぎると認知負荷が上がり、少なすぎると訓練密度が下がる。

### 4.4 段階的ヒントの設計

再生制御は以下のルールで段階的に解放される：

| 状態 | 解放されている操作 |
|---|---|
| 初期 | 自然速度（Lv 設定通り）の再生のみ |
| 2回再生後 | 0.75x スロー再生ボタンが追加で解放 |
| 5回再生後（将来拡張） | スクリプトプレビューが解放（未実装） |

無限リプレイを排除しているわけではなく、**「リプレイ回数を可視化することで学習者の自己認識を促す」**設計。replay カウンタは常に表示される。

### 4.5 スコアリングの許容範囲

| モード | 判定基準 |
|---|---|
| Cloze | 各空欄を個別に判定。大文字小文字・末尾の句読点は無視。`gonna` ↔ `going to` のような等価形は正解扱い |
| Full Dictation | 単語レベルの Levenshtein 距離で `accuracy = 1 - edits/totalWords` |
| Minimal Pair | 完全一致のみ正解 |

Cloze の等価形テーブル（`src/lib/scoring.js` の `EQUIVALENTS`）：

```
gonna ↔ going to
wanna ↔ want to
gotta ↔ got to
lemme ↔ let me
gimme ↔ give me
kinda ↔ kind of
sorta ↔ sort of
dunno ↔ don't know
```

これらの等価形を許容するのは、**「縮約を聞き取れたか」を判定したいのであって、「正書法でどう書くか」を問うているのではない**から。学習者は `gonna` と書いても `going to` と書いても、層3の縮約を捕捉できていれば正解。

### 4.6 声と対話の設計

| 役割 | 声 | 性別 | 理由 |
|---|---|---|---|
| 話者 A（デフォルト） | `nova` | 女性 | 明瞭で中庸、CEFR A2–B1 教材に多い声質 |
| 話者 B（対話時） | `onyx` | 男性 | 性別差で話者交替を認識しやすい |

同性ペアは認知負荷が一段上がるため Lv5 では採用しない。将来 Lv6 を導入する場合の拡張余地として残す。

対話 MP3 の連結は GAS 側で行ごとに別 API 呼び出し → MP3 バイト連結。`gpt-4o-mini-tts` が単一話者しか扱えない制約への回避策。

### 4.7 キャッシュキーの組成

```
sha256(voice | speed | instructions | text)
```

`text` のみではなく `voice` `speed` `instructions` も含めるのは：

- 同じ文でも声・速度・スタイル指示が異なれば別音声になるため
- キャッシュヒット率は下がるが、誤って違うパラメータの音声を返すリスクを排除

`instructions` は Claude が文ごとに生成するため、同じ文でも文脈（シーン）が違えばキャッシュは別になる。意図通り。

将来、キャッシュサイズが膨らんだ場合は LRU 削除を GAS 側に実装する。

### 4.8 例文生成プロンプト（Claude API）の設計

#### 思想

Claude への生成プロンプトは本アプリの**心臓部**である。ここで生成される文の質が、その後の TTS・ディクテーション・診断のすべての精度を律する。プロンプト設計で守るべき原則：

1. **`target_features` は ground truth として扱う**
   診断ロジックは Claude が返した `target_features` を正解として用いる。逆に言えば、Claude が嘘の features を返すと診断が静かに破綻する（学習者には「捕捉した」と表示されているが、実際にはその現象は文に含まれていない）。プロンプトで「features は実際の文に含まれる現象のみを記述すること、捏造してはならない」を強く言い切る必要がある。

2. **出力は厳格な JSON**
   余計な前置き・コードフェンス・コメントを混入させない。`{ ... }` 以外の文字が混じるとパース失敗で1問丸ごと失う。`system` プロンプトと末尾の `Output JSON only. No backticks.` で二重に縛る。

3. **語彙制約は強く・具体的に**
   「易しい単語で」では弱い。「CEFR A2–B1」と明示し、それでも高度な語が出てきたらプロンプトを締める。**語彙ノイズはあらゆる診断の精度を下げる**ため最優先で抑える。

4. **シーンの説明は具体的に**
   「電話」だけだと汎用すぎる。「customer support, booking, appointments」のように何の電話か例示する。Claude は抽象指示には平凡な答え、具体指示には個別性のある答えを返す。

5. **レベル間の差は知覚可能でなければならない**
   Lv1 と Lv2 で「説明上の違い」があっても、生成される文が音韻的に区別不可能なら難易度設定は無意味。「Lv1 は no contractions」「Lv2 は standard contractions OK」のように、**文字列レベルで違いが現れる制約**を入れる。

6. **シーン × レベルの整合性を担保する**
   電話の Lv1 は「customer service の formal な定型句」、店舗の Lv4 は「カフェ店員の砕けた一言」というように、シーンとレベルの組み合わせごとに自然な発話像がある。プロンプトはその像を明示的に伝える。

#### 具体（現行プロンプトの構造）

`src/lib/prompts.js` の `buildGenerationPrompt` は以下の構造で組み立てている：

1. **シーン記述**: `SCENES[scene].description` で場面を具体的に例示
2. **レベル記述**: `LEVELS[level].rules` で文長・縮約・連結の必須条件を指定
3. **対話の有無**: Lv5 のみ `lines` 配列を 2–4 ターンで返させる
4. **モード固有指示**: Cloze なら `blanks` 配列、Minimal Pair なら `minimal_pair_target` を必須化
5. **target_features カタログ**: 使える feature トークンを限定列挙
6. **JSON スキーマ**: 厳密に定義し、末尾で `Output JSON only` を強調

#### 改修時に注意すべきポイント

| 観点 | 注意点 |
|---|---|
| プロンプトの肥大化 | 制約を増やしすぎると Claude の創造性が落ち、似た文ばかり出る。逆に緩めすぎると診断価値の低い文になる。バランスを見ながら微調整 |
| feature 分類の追加 | 新 feature（例：`assimilation:did_you`）を増やすときは、(1) カタログに追加、(2) Cloze 空欄選定ロジックで認識、(3) スコアリング側で診断対応、の3点セット |
| パターン固着 | temperature 0.9 でも同じ構文（"I was gonna..." 等）が繰り返される。気付いたらプロンプトに「avoid starting with 'I was'」のような明示的多様性指示を追加 |
| 文化的偏り | プロンプトに何も書かないと米国寄り。Pines（フィリピン）滞在中という Naoya の状況からは "neutral international English" 指定も選択肢 |
| 不適切内容の混入 | 職場シーンでの政治・人事評価、店舗での対立的やり取り等、学習集中を削ぐ題材は `topics: everyday, neutral, non-controversial` で抑制 |
| モデル変更時 | `claude-haiku-4-5` から別モデルに変えたら、プロンプトの効き方が変わる。20 件ほど出力サンプルを目視で検品してから本番化 |

#### よくある失敗モードと対策

| 症状 | 原因 | 対策 |
|---|---|---|
| `target_features` の現象が実は文中にない | Claude が後付けで features を生成 | プロンプトで「features は actual content の要約であり、捏造してはならない」と明示 |
| Lv4 なのに縮約が出ない | "allowed" では使われない | "must include at least one casual reduction" と必須化 |
| Cloze の `blanks` が target_features と無関係な語 | Claude が空欄選定方針を理解していない | `blanks should correspond directly to target_features where possible` を modeSpecific に追記 |
| Minimal Pair の distractor が易しすぎる | "wrong words" では曖昧 | `phonologically confusable for Japanese learners (r/l, θ/s, ɪ/iː, æ/e, f/h)` を強調 |
| 全角文字や絵文字が混入 | TTS・Cloze 入力で予期せぬ挙動 | プロンプトで `ASCII only in English fields` |
| 翻訳が直訳調 | `translation_ja` の指示が機械的 | `Japanese translation that sounds natural to a native speaker` を追加 |

### 4.9 音声合成プロンプト（OpenAI gpt-4o-mini-tts）の設計

#### 思想

TTS への指示は、Claude が生成した文の**音韻的意図を音声として実体化する**役割を持つ。ここで指示が不適切だと、Claude が `reduction:gonna` を含めたつもりの文を、TTS は `going to` と過剰明瞭に発音し、訓練価値が消失する。

守るべき原則：

1. **Claude の意図と TTS の挙動を整合させる**
   Claude が `target_features` にどんな現象を入れたかを、TTS の `instructions` にも反映させる。具体的には Claude 自身に `tts_instructions` フィールドを返させ、それをそのまま TTS に渡す。これにより「言いたい現象」と「実際に発音される現象」のずれを最小化する。**この整合が本アプリ最大の品質指標**。

2. **声の personality は強い**
   `nova` は明るく親しみやすい、`onyx` は深く落ち着いた、`alloy` は中庸、`shimmer` はやわらかい等、ボイスごとに既定の話者像がある。シーンと整合しないと不自然（例：`onyx` で店員役は重すぎる）。デフォルトは `nova`、対話の話者B のみ `onyx`。

3. **`speed` は base、`instructions` は modulation**
   `speed: 1.0` は基本値で、`instructions: "speak clearly and slowly"` を組み合わせると体感的にはさらに遅く感じる。レベルごとの目標速度感は両者の組み合わせで作る。`speed` だけで全てを表現しようとしない。

4. **過剰な感情・キャラ付けは避ける**
   `"excited"` `"dramatic"` のような指示は学習を邪魔する。**自然な日常会話のレジスター**が望ましい。例外は Lv5 対話で話者性を出したいときのみ、それも控えめに。

5. **同一話者は同一ボイス**
   対話で話者Aが2回登場するなら、両方とも同じボイスで生成する。途中で切り替えると学習者が話者識別を学習できない。GAS 側で `line.speaker === 'B' ? voiceB : voiceA` という単純なロジックでこれを保証している。

6. **綴りは音声化される**
   TTS は綴り通り読む傾向が強い。`gonna` と書けば `gonna` と発音し、`going to` と書けば `going to` と発音する。Claude の出力で文中に `gonna` が含まれていることが、TTS で縮約が再現される第一条件。

#### 具体（現行の instructions 設計）

現状、`tts_instructions` は Claude が文ごとに動的生成する設計になっている（`prompts.js` で要求）。例えば：

- 電話 × Lv2: `"Friendly customer-service tone, slightly polite, natural pace, mild linking."`
- 店舗 × Lv4: `"Casual cafe staff voice, natural conversational pace, with relaxed reductions."`
- 職場 × Lv3: `"Neutral colleague tone, natural pace, clear but not over-articulated."`

これにより文ごとにシーン適合の指示が動的に変わる。

#### レベル別の理想的 instructions パターン

参考として、各レベルで目指す指示の雰囲気：

| Lv | speed | instructions の方向性 |
|---|---|---|
| 1 | 0.85x | `"Read clearly and slowly, articulate every word, textbook-like."` |
| 2 | 0.9x | `"Relaxed natural pace with standard contractions, but no further reductions."` |
| 3 | 1.0x | `"Natural conversational pace with normal linking between words. Do not over-articulate."` |
| 4 | 1.05x | `"Natural conversational pace, with relaxed casual reductions where typical."` |
| 5 | 1.05x | `"Speak naturally with the personality of the speaker. Use casual reductions."`（話者A・Bで性格差を出してもよい） |

これらは固定値ではなく、Claude が文ごとに微調整した上で `tts_instructions` に返す。Claude プロンプト側に上記の方向性を仕込むことで、文内容と整合した instructions が動的生成される。

#### 改修時に注意すべきポイント

| 観点 | 注意点 |
|---|---|
| instructions の長さ | 1文・10–25 語程度が最適。長すぎると TTS が指示自体に引きずられる。短すぎると指示が効かない |
| 矛盾する指示の回避 | `"slowly and clearly"` + `"with natural reductions"` は矛盾。レベルとシーンが噛み合うよう Claude 側で整合性チェックを促す |
| 過剰な明瞭化の抑制 | TTS のデフォルトは「教科書的明瞭発音」に寄りがち。Lv3 以上では `"do not over-articulate function words"` を明示的に縛る |
| 縮約の保持 | `gonna` を `gonna` のまま発音させたいなら、文の綴りを `gonna` で固定する（Claude 出力時点で）。`going to` に置換しない |
| アクセントの統一 | 指示しないとアメリカ英語寄り。意図的に British / Neutral にしたい場合は明示 |
| ピッチや感情の指定 | `instructions` でピッチ・感情も操作可能だが、本アプリは「自然な日常会話」を基本姿勢とするため感情強調は原則しない |
| ボイス追加時 | OpenAI が新ボイスを追加した場合、その personality を聴き比べてから採用判断する。`coral` `verse` 等は感情表現が強く、本アプリ向きでないものもある |

#### よくある失敗モードと対策

| 症状 | 原因 | 対策 |
|---|---|---|
| Lv3 以上なのに連結が起きていない | TTS が単語境界を強調 | instructions に `"with natural linking, blend words smoothly"` |
| `gonna` が `going to` と発音される | TTS が綴りを正規化、または instructions が "speak clearly" 寄り | (1) 文中の綴りを `gonna` で固定、(2) instructions に `"keep reductions as written"` |
| 対話の話者A・Bが同じ声に聞こえる | 同性ペアか声質が似ている組み合わせ | デフォルトの nova（女）× onyx（男）から外れないようにする |
| 不自然に重々しい / 芝居がかった | キャラ付け指示が強すぎる | `"emotional"` `"dramatic"` を排除、`"neutral, conversational"` を基準語に |
| 同じ文でも毎回違う音声 | キャッシュキーが正しく機能していない | `sha256(voice\|speed\|instructions\|text)` の4要素が全て一致しているか確認 |
| 対話の話者間の間（ま）が不自然 | 単純バイト連結のみ実装、無音フレームなし | 将来 GAS 側で短い無音 MP3 を間に挿入する拡張 |

#### Claude × OpenAI ハンドオフの整合性

この2つの API は独立して動くが、**学習体験としては1つの統合された出力**として現れる。両者の整合を保つには：

- Claude プロンプトで `tts_instructions` フィールドの生成ルールを明文化する（現状の `prompts.js` がそれを担う）
- Claude が生成した文の `target_features` と、TTS が実際に発音した結果がずれていないか、**プロンプト変更後は必ず抜き取り検品する**
- 検品ワークフロー:
  1. Claude で 10 件生成
  2. TTS で音声化
  3. Naoya が聞いて features が実体化しているか主観評価
  4. ずれが多ければプロンプトを締め直す
- 検品結果は本書末尾の付録（将来追加）に蓄積し、プロンプト改修の根拠とする

**このハンドオフ品質こそ本アプリの実質的な品質指標**であり、表面的な「文の自然さ」よりも重要。Claude 単体・TTS 単体で個別に評価しても、この整合は測れない。

## 5. 意図的にスコープアウトした要素

| 要素 | 理由 | 補完先 |
|---|---|---|
| 産出側（発音・流暢性） | 入力専用アプリとして焦点を絞る | Pines Academy 1-on-1 |
| イントネーション学習 | テキスト化でピッチ情報が失われ判定不能 | 添付 Speaking Training 教材 Part1・2 |
| 内容理解問題（comprehension） | 推測で正答可能で診断精度が低い | TED-Ed / VOA Learning English |
| 文法解説 | リスニングアプリの責務外 | English-Structure-Trainer |
| 語彙拡充 | 同上 | English-Questions / English-Phrase |
| 進捗の永続化 | プロトタイプではブラウザ localStorage のみ | 将来拡張（4.8 参照） |
| 多人数（3人以上）対話 | 認知負荷過剰、TTS の自然さも低下 | 不要と判断 |
| 騒音背景・実環境録音 | TTS では再現できない | 別の実音声教材で補完 |

「これも入れた方がいいのでは」という改修提案が出たとき、この表に該当する場合はまず **「補完先で扱うべき」と判断できないか** を検討する。本アプリが何でも屋になることを防ぐ。

## 6. 改修方針

### 6.1 改修判断のフロー

新機能の追加判断は以下の順で：

1. その機能は層3（あるいは副次的に層2・4）を強化するか？
   - No → 慎重に。本アプリの存在意義から外れる
2. その機能は診断精度を上げるか？それとも完遂量を上げるだけか？
   - 完遂量だけなら不要
3. その機能は構造化フィードバックに繋がるか？
   - No → なぜ追加するのか再考
4. その機能を追加することで UI の「静けさ」が崩れないか？
   - 崩れるなら、別の方法を探す

### 6.2 ドキュメント階層

| ドキュメント | 役割 | 編集権限 |
|---|---|---|
| `docs/background.md`（本書） | なぜそうなっているか・思想 | Claude（Naoya の合意のもと） |
| `docs/specification.md` | 何ができるか・データ構造 | Claude（Naoya の合意のもと） |
| `README.md` | セットアップ・運用 | Claude |
| `src/lib/prompts.js` | 文生成の具体的制約（実装レベルの仕様） | Claude → Cursor |

`docs/` 以下を Cursor が変更する場合は、Naoya からの明示的な指示を必要とする（thinkgrindai の CLAUDE.md と同じ運用）。

### 6.3 将来の拡張候補

優先度の高い順：

1. **進捗の永続化と弱点トラッキング**: GAS + Google Sheets に feature 別の正答率を蓄積。「`weak_form:to` の捕捉率 65%」のような可視化
2. **スパース・リピート出題**: 弱点 feature を含む文を優先的に生成するよう、生成プロンプトに動的に弱点情報を渡す
3. **IPA 表示オプション**: Review 画面で、聞き落とした語の IPA をオプション表示
4. **シーン拡張**: 「友達との雑談」「カジュアルな職場会話」を追加（TTS の自然さを再評価したうえで）
5. **Lv6（同性ペア対話）**: 話者識別の認知負荷を一段上げる
6. **音素レベルのミニマルペア訓練モード**: 単語ではなく音節レベルで `/r/` `/l/` を聞き分けるドリル

各拡張は本書のセクション3（思想）と整合するかを確認したうえで実装する。

## 7. 用語集

| 用語 | 定義 |
|---|---|
| 層3 | 添付 `english_speech_structure.md` の機能軸5層のうち「連結部」。リンキング・弱形・脱落・縮約・同化を含む |
| 弱形（weak form） | 機能語が文中で弱く短く発音される現象。`to → /tə/`, `and → /ən/` |
| リンキング（linking） | 単語境界で前の子音と次の母音が繋がる現象。`pick it up → pi-ki-tup` |
| 縮約（reduction） | 口語で語彙化された短縮形。`going to → gonna`, `did you → didja` |
| 脱落（elision） | 環境依存で音が落ちる現象。`next day → nex-day` |
| ミニマルペア | 1音だけ違って意味が変わる語の対。`right / light`, `ship / sheep` |
| target_features | Claude が文生成時に返す、その文に含まれる層3現象のタグ配列 |
| CEFR | 言語能力の国際指標。A2＝初中級、B1＝中級 |
